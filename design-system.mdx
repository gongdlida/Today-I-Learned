# 디자인 시스템

UI 결정을 일관되게 내리기 위한 ‘규칙 + 언어 + 기준’의 집합 (결과물이 아니라 판단 체계)

## 목적

표면적인 목적은 UI 통일, 컴포넌트 재사용성, 시각적 일관성 향상
하지만 궁극적인 목적은 UI 의사결정을 사람 -> 시스템으로 이전 (디자인 **시스템**)

- 사람이 바뀌어도 기준은 유지되어야 함
- “이건 디자이너 취향인가?”라는 질문 제거
- 논쟁 비용 감소

## 디자인 시스템의 효과

디자인 시스템의 단기적 효과는 거의 없다고 볼 수 있다.
오히려 초기 구축과 유지보수에 비용이 든다.
그래서 초기에 시스템을 구축하는 과정에서 반감이 생겨 흐지부지되는 경우가 많다.

하지만 장기적으로 보면 다음과 같은 효과가 있다:

- UI 변경 비용 감소
- 예외 상황 대응 속도 증가
- 조직 내 UI 암묵지 제거
- 신규 인력 온보딩 가속

디자인 시스템은 속도를 높이는 도구가 아니라, 속도를 ‘잃지 않게’ 만드는 장치

## 디자인 시스템의 범위를 오해하는 경우 발생하는 문제

잘못된 이해

- 모든 UI를 통제해야 한다.
- 예외를 허용하면 안 된다.

올바른 이해
디자인 시스템은 예외를 없애는 시스템이 아닌 예외를 설명 가능하게 만드는 시스템

- “이건 왜 다르죠?” → 설명 가능
- “그냥 그렇게 했어요” → 실패

## 디자인 시스템 vs 관련 개념 구분

UI Kit = 시각 자산 모음
Component Library = 코드 레벨 재사용
Style Guide = 시각 규칙 문서
Design System = 위 모든 것을 관통하는 판단 체계

## 디자인 시스템 구축 시 유의점

내가 경험한 조직에서 대부분은 스타일 가이드와 UI Kit만 존재하고,  
왜 그렇게 사용하는지에 대한 가이드가 없다보니,
디자인 시스템이 있다고 착각했기 때문에 디자이너가 변경될 때마다 UI가 달라지는 문제가 발생했다.

이로인해 다음과 같은 문제가 발생했다:

- 어제까지 맞던 UI가 오늘은 틀림
- “시스템 바뀌었어요” 한마디로 대규모 수정
- 근거 없는 변경 → 코드 안정성 붕괴

그래서 디자인 시스템은 믿을 수 없다는 인식이 생겼고,  
결국 많은 리소스를 들여 만든 디자인 시스템은 무용지물이 되었다.

디자인 시스템 운영에 실패한 원인은 UI 의사결정을
시스템으로 하는 것이 아닌 사람 중심으로 했기 때문이다.

디자인 시스템 구축 시 시각적인 것뿐만 아니라
판단 체계(규칙 + 언어 + 기준)를 함께 정의하는 것이 중요하다는 점을 잊지 말자.
